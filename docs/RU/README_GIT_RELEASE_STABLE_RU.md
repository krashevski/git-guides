# Выпуск стабильной версии (release)

[🇬🇧 English](../EN/README_GIT_RELEASE_STABLE_EN.md) | [🇷🇺 Русский](README_GIT_RELEASE_STABLE_RU.md)

## 1️⃣ Подготовка к выпуску

1. Перейти в основную ветку разработки
Обычно `main` или `master`:
```bash
git checkout main
git pull origin main
```

2. Проверить статус и убедиться, что нет незакоммиченных изменений
```bash
git status
```

3. Убедиться, что все изменения протестированы
* Запустить тесты, линтеры, сборку.

* Исправить баги.
  Стабильный релиз не должен ломать существующую функциональность.

## 2️⃣ Создание ветки для релиза (опционально)

Если вы хотите подготовить стабильную ветку отдельно:
```bash
git checkout -b release/stable
```

* Это удобно, если вы делаете исправления багов перед выпуском.
* После подготовки изменений ветку можно слить в `main`.

## 3️⃣ Обновление версии

1. Измените файл с версией (например VERSION или package.json) на нужную:
```text
1.2.3
```

2. Закоммитьте изменения:
```bash
git add VERSION
git commit -m "Bump version to 1.2.3 for stable release"
```

[I] Если нет скрипта VERSION, то можно сохранять историю изменений в скрипте CHANGELOG.md

## 4️⃣ Тегирование стабильной версии

В Git принято использовать аннотированные теги для релизов:
```bash
git tag -a stable -m "Stable release for git-guides"
```

> [I] Если хотите, можно также поставить конкретную версию:
> [I] git tag -a v1.2.3 -m "Release v1.2.3 stable"

Проверить тег:
```bash
git tag
```

## 5️⃣ Публикация ветки и тега на удалённый репозиторий

```bash
git push origin main      # или release/stable
git push origin stable    # пушим тег stable
```

Теперь тег `stable` доступен на GitHub/GitLab.

## 6️⃣ Проверка на удалённом репозитории

1. Список всех тегов:
```bash
git fetch --tags
git tag -l
```

2. Проверка изменений, связанных с тегом `stable`:
```bash
git log stable
```

## 7️⃣ Поддержка и хотфиксы

Если после релиза обнаружены баги:
1. Создайте ветку для хотфикса:
```bash
git checkout -b hotfix/fix-typo stable
```

2. Внесите исправления, закоммитьте, обновите тег или создайте новый:
```bash
git commit -am "Fix typo in guides"
git tag -f stable  # перезаписываем тег stable на исправленную версию
git push origin hotfix/fix-typo
git push origin stable --force
```

[!] ⚠️ Использовать --force аккуратно, чтобы не ломать локальные копии других участников команды.

## Итог процесса

1. Подготовка и тесты → main
2. (Опционально) Создание release branch
3. Обновление версии
4. Создание аннотированного тега stable
5. Публикация ветки и тега
6. Проверка на удалённом репозитории
7. Хотфиксы и обновление тега при необходимости
