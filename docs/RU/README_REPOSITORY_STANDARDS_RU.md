# 📘 Руководящие принципы по управлению репозиторием (стандартизация)

[🇬🇧 English]( ../EN/README_REPOSITORY_STANDARDS_EN.md) | [🇷🇺 Русский](README_REPOSITORY_STANDARDS_RU.md)

Этот документ определяет рекомендации по поддержке репозитория, ветвлению, коммитам, тестированию и релизам.
Цель — поддержание высокого качества кода, стабильной основной ветки и прозрачности разработки.

## 1. Ветвление

### 1.1 Ветка main и другие ветки
* main — стабильная, безопасная ветка.
Содержит окончательный, функциональный, полностью протестированный код.

* develop (необязательно) — промежуточная ветка для интеграции функций и подготовки релизов.
Если проект небольшой, ветку develop можно опустить.


### 1.2 Ветви разработки функций
Каждая задача реализуется в отдельной ветке:
```bash
feature/<краткое название задачи>
```

Примеры:
- feature/language-switch
- feature/disk-check-improvements
- feature/messages-refactoring

### 1.3 Ветви исправления ошибок
Исправления ошибок:
```bash
bugfix/<краткое описание>
```

Примеры:
bugfix/mount-detection
bugfix/menu-return-loop

### 1.4 Ветви релизов
- Если используется develop, перед релизом создается следующее:
```bash
release/<версия>
```

- В противном случае релизы собираются непосредственно из main.


### 1.5 Резервные ветки
Перед внесением критических изменений создается защитная ветка:
```bash
backup/<дата или имя задачи>
```

Пример:
```bash
backup/2025-12-10-pre-refactor
```

## 2. Правила коммитов

### 2.1 Формат сообщения коммита
Используйте содержательный, краткий и понятный формат:
```bash
<type>: <short description>
```

Типы:
- feat — новая функция
- fix — исправление
- refactor — улучшения внутренней логики
- docs — изменения в документации
- test — тесты
- build — сборка/структура проекта
- style — незначительные изменения без изменения логики
- chore — другое обслуживание

Примеры:
```bash
feat: добавить переключение языка в menu.sh
fix: исправить обнаружение монтирования USB в show-system-mounts.sh
docs: обновить рабочий процесс git раздел
refactor: разделение messages.sh на модули
```

### 2.2 Требования к публикации коммита
1. Коммит содержит завершённое изменение.
2. Большие изменения следует группировать в один логический коммит, чтобы история оставалась чистой и читаемой.

## 3. Правила слияния

### 3.1 Слияние с main
1. Разрешено только после полного тестирования (см. раздел 5).
2. Слияние выполняется с помощью команды:
- merge — сохраняет полную историю коммитов
- squash merge — объединяет все коммиты в один

> **[I] Squash**
> - squash удобен, если была черновая работа или много небольших коммитов типа «исправление» или «в процессе работы» в рамках функции.
> - squash создает один коммит в main, что делает историю чище.

3. Отправка изменений в main запрещена; все изменения должны проходить через ветки функций и тестирование.

### 3.2 Как обновить основную ветку из ветки разработки
1. Обновите ветку main последними изменениями из ветки разработки:
```bash
git checkout main
git pull origin main
git merge feature/xxx
git push origin main
```

2. Устраните конфликты.
3. Протестируйте пакет локально.
4. Выполняйте слияние с веткой main только после тестирования.
5. Удаляйте ненужные ветки для поддержания чистоты репозитория:
```bash
git checkout main
git pull
git branch -d feature/xxx
git push origin --delete feature/xxx
```

## 4. Процесс выпуска

### 4.1 Нумерация версий
Используется схема SemVer:
```bash
MAJOR.MINOR.PATCH
```

Примеры:
- 1.0.0 — первый стабильный релиз
- 1.1.0 — новая функция
- 1.1.1 — исправление ошибки

### 4.2 Создание релиза
1. Код в main должен быть полностью протестирован.
2. Компилируется список изменений.
3. Создайте git-тег:
```bash
git tag -a v1.2.0 -m "Release 1.2.0"
```

4. Отправьте тег:
```bash
git push origin v1.2.0
```

## 5. Требования к тестированию перед публикацией

Изменения должны быть протестированы на локальном компьютере перед отправкой в ​​основную ветку.

### Смотри также
- Тестирование прежде merge в main [README_GIT_TESTING_RU.md](README_GIT_TESTING_RU.md)

## 6. Стиль кода с открытым исходным кодом
Код должен быть удобочитаемым, содержать четкие имена переменных, аккуратный формат и включать комментарии там, где это необходимо.

## 7. Резюме

Этот стандарт обеспечивает:
- Стабильность основной ветки.
- Профессиональное качество проекта.
- Снижение риска регрессий.
- Запланированные релизы.
